Schema JSON Specification

This guide outlines the structure and requirements for the schema.json files used throughout the Data Registry. These files provide the metadata necessary for generating interactive Entity Relationship Diagrams (ERDs) and detailed table documentation.

Root Object: Schema

Section titled “Root Object: Schema”

The top-level object that defines the entire data model.

  • name: The name of the schema.
  • desc: A description of what this schema represents.
  • tables: An array of Table objects (Required).
  • relations: An array of Relation objects.
  • functions: An array of Function definitions.
  • enums: An array of Enum definitions.
  • driver: A Driver object containing database engine details.
  • labels: Global labels/tags for the schema.
  • viewpoints: An array of Viewpoint objects for curated diagram views.

Constraints

Section titled “Constraints”

Defines rules enforced on the data (Primary Keys, Foreign Keys, etc.).

  • name: Constraint identifier.
  • type: The type (e.g., “PRIMARY KEY”, “FOREIGN KEY”, “UNIQUE”).
  • def: The SQL definition or raw text of the constraint.
  • table: The table this constraint belongs to.
  • referenced_table: For foreign keys, the table being referenced.
  • columns: Array of columns involved in the constraint.
  • referenced_columns: Array of columns being referenced (for foreign keys).

Indices

Section titled “Indices”

Optimization structures for data retrieval.

  • name: Index name.
  • def: The SQL definition used to create the index.
  • table: The table the index is built on.
  • columns: Array of columns covered by the index.

Enums

Section titled “Enums”

Custom data types with a fixed set of values.

  • name: The name of the enumeration.
  • values: Array of strings representing the allowed values.

Functions

Section titled “Functions”

Database-level logic or procedures.

  • name: Function name.
  • return_type: The data type returned by the function.
  • arguments: String representation of function parameters.
  • type: The type of function (e.g., “trigger_function”, “scalar”).

Organizational Metadata

Section titled “Organizational Metadata”

Labels

Section titled “Labels”

Lightweight tags and metadata for categorization.

  • name: The label text.
  • virtual: Boolean. If true, indicates the label is logical/calculated rather than physical.

Viewpoints

Section titled “Viewpoints”

Curated subsets of the schema for specific documentation needs.

  • name: Viewpoint title.
  • desc: Purpose of this view.
  • tables: Array of table names to include in this view.
  • distance: Number of relationship degrees to include from the primary tables.
  • groups: Array of Viewpoint Groups.

Viewpoint Groups

Section titled “Viewpoint Groups”

Visual grouping of tables within a specific Viewpoint.

  • name: Group title.
  • desc: Description of the group’s purpose.
  • tables: Tables belonging to this visual cluster.
  • color: Hex or CSS color used to highlight this group in diagrams.

System Context

Section titled “System Context”

Driver

Section titled “Driver”

Metadata about the database engine and version.

  • name: Driver name (e.g., “sqlite”, “postgres”).
  • database_version: The version of the source database.
  • meta: Advanced metadata object containing current_schema, search_paths, and engine-specific dictionaries.
